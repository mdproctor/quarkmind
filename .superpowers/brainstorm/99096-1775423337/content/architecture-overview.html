Overall Architecture

Single Quarkus module · Four package layers · Plugin seams at the intelligence layer

📡 sc2 package — SC2 I/O Adapter
StarCraftClient
manages SC2 connection via ocraft-s2client
GameObserver
receives per-frame SC2 observations
CommandDispatcher
sends SC2 commands back each frame
▼ raw SC2 observations                                          SC2 commands ▲
🗺 domain package — Game State Model
GameState
master snapshot each decision cycle
Unit · Building · Resource
plain Java domain objects
GameMap · PlayerInfo
terrain, fog of war, enemy intel
▼ structured GameState                                       high-level intents ▲
🧠 agent package — CaseHub Intelligence Layer
StarCraftCaseFile
CaseFile with typed SC2 keys
GameStateTranslator
GameState → CaseFile updates
CaseEngine (CaseHub)
control loop, drives TaskDefinitions
IntentQueue
collects actions for CommandDispatcher
🔌 Plugin seams (AgentTask implementations)
StrategyTask
what strategy are we executing?
EconomicsTask
resources, workers, supply, builds
TacticsTask
army, combat, micromanagement
ScoutingTask
map awareness, enemy intel
▼ dummy implementations (Phase 1)      future: Drools · Quarkus Flow · CaseHub nested ▲
🧩 plugin package — Default Implementations
PassThroughStrategyTask
fixed strategy, no logic
BasicEconomicsTask
just keep workers mining
PassThroughTacticsTask
units do nothing
PassThroughScoutingTask
no scouting
Key constraint: All layer boundaries are CDI interfaces. Swap a plugin by providing a new @ApplicationScoped bean — no wiring changes. Native Quarkus compatible by design.
