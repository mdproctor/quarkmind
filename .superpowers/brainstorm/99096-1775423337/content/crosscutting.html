Cross-Cutting Concerns

Issues that touch every layer — need to be decided before or during Phase 1

⏱ Threading & Timing Bridge

SC2 game loop is synchronous — ocraft-s2client calls your bot every frame (~22fps) and waits for commands. CaseEngine is async. These two clocks need a clear bridge.

Proposed: GameObserver translates frames to CaseFile updates at SC2 speed. CaseEngine runs on its own schedule (event-driven on CaseFile changes). An IntentQueue buffers actions; CommandDispatcher drains it each frame. The two loops never block each other.
🏗 Native Quarkus Guardrails

Every plugin added risks drifting away from native compatibility. Need explicit rules enforced by convention, not hope.

Rules:
• No dynamic class loading or runtime code generation
• All CDI injection via constructor or field with proper annotations
• Reflection uses → register in reflection-config.json
• ocraft-s2client (RxJava + Protobuf) needs native verification before going far
• A NATIVE.md tracks known issues and workarounds per dependency
🧪 SC2-Free Testability

R&D iteration speed requires testing plugin logic without launching StarCraft II. Connecting to a live game just to test strategy logic is too slow.

Proposed: SimulatedGameObserver — a test-mode alternative to the real SC2 adapter that injects scripted GameState snapshots. Activated via Quarkus profile (%test). Intelligence layer is completely unaware of the difference.
🎮 Race Selection

The domain model (units, buildings, production chains) is heavily race-specific. Choosing a race shapes the entire domain package from day one.

Recommendation: Protoss.
Simplest production model (no larvae/Zerg mechanics, no reactor/tech-lab/Terran floating buildings). EISBot used Protoss for the same reason. Domain model starts smallest, easiest to fill in over time.
⚖️ Action Conflict Resolution

Multiple TaskDefinitions (e.g. Economics + Tactics) can both want to command the same worker unit. Without resolution, commands conflict or overwrite.

Proposed: IntentQueue uses typed intents (BuildIntent, AttackIntent, MoveIntent) with unit ownership. CaseEngine's PlanningStrategy arbitrates conflicts before dispatch. Priority: Tactics > Economics > Strategy for unit commands.
📊 Observability

This is an R&D project — understanding why the bot made a decision is as important as the decision itself. CaseHub's PropagationContext already supports this.

Proposed: Structured logging on every CaseFile write and every TaskDefinition activation. Quarkus Dev UI visible during development. Can wire into full OpenTelemetry later via the quarkus-observability skill.
Most critical to get right in Phase 1: The threading/timing bridge and native guardrails. Getting these wrong early creates compounding problems. The others can evolve.
