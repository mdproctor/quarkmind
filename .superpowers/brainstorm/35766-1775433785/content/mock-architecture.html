Mock-First SC2 Architecture

Full stack developed and tested against a stateful mock SC2 — real SC2 slots in as an alternative implementation

📡 sc2 package — CDI interfaces (the contract)
SC2Client
connect / join game / leave game
GameObserver
produces GameState each cycle
CommandDispatcher
drains IntentQueue → commands
implements ▼
🧩 sc2.mock — Stateful Mock (Phase 0+)
MockSC2Client
simulates connect/join/leave lifecycle
MockGameObserver
generates GameState from SimulatedGame
MockCommandDispatcher
applies intents → mutates SimulatedGame
SimulatedGame (internal state)
units & positions resources supply build queue game clock
ScenarioLibrary — named mutations:
spawn_enemy_attack · set_resources · enemy_expand · supply_almost_capped
🎮 sc2.real — ocraft-s2client (Phase 1+)
RealSC2Client
ocraft-s2client connection + lifecycle
RealGameObserver
translates SC2 protobuf observation → GameState
RealCommandDispatcher
translates intents → SC2 protobuf commands
SC2 Debug API (integration QA)
ocraft-s2client exposes SC2's debug interface — same named scenarios from ScenarioLibrary triggered against live game for integration parity.
ScenarioLibrary has two implementations:
MockScenarioRunner → mutates SimulatedGame
SC2DebugScenarioRunner → calls SC2 Debug API
Profile / CDI switching
%mock (default) → sc2.mock beans active
%sc2 → sc2.real beans active
%test → mock + scripted scenarios
Intelligence layer (CaseHub, plugins) is completely unaware of which SC2 implementation is active.
♻ The living specification loop
Real SC2 surprises us → Update SimulatedGame to replicate the quirk → Write a test that would have caught it → CI catches this class of problem forever after
